## 1. What Judges Look For

### Core Judging Criteria (Universal)
Judges at AI agent hackathons consistently evaluate across these dimensions:

| Criterion | Weight | What They Want to See |
|-----------|--------|----------------------|
| **Innovation/Originality** | High | Novel approach, not a tutorial clone. Unique angle on a real problem. |
| **Technical Implementation** | High | Clean architecture, working demo, smart use of tools/APIs. |
| **Real-World Impact** | High | Solves a genuine pain point. Could this become a product? |
| **User Experience** | Medium | Intuitive, polished UI even if minimal. Smooth demo flow. |
| **Completeness** | Medium | Core functionality works end-to-end. Not half-built. |
| **Presentation Quality** | High | Clear pitch, compelling narrative, concise demo. |
| **Use of Required/Sponsor Tech** | Varies | Deep, meaningful integration (not superficial). |

### What Separates "Good" from "Winner"
- **Transparency & Explainability**: Winners show reasoning traces, decision logs, and make their agents' behavior interpretable rather than black-box.
- **Production Readiness**: Judges increasingly favor projects that look like they could plug into real enterprise workflows, not just toy demos.
- **Wow Factor**: A simple, memorable moment in the demo that makes judges think "I want to use this."

### Judge Psychology
- Judges see 20-50+ projects. They are tired, distracted, and overwhelmed.
- They remember stories and emotions, not technical specs.
- Check judges' backgrounds beforehand -- if they're VCs, emphasize market opportunity; if they're engineers, show architectural elegance; if they're from a sponsor, show deep integration with their product.

## 2. Winning Patterns from Recent AI Agent Hackathons

### Pattern 1: Multi-Agent Collaboration Systems
Winners increasingly build systems where multiple specialized agents coordinate rather than a single monolithic agent.

**Examples:**
- **RiskWise** (Microsoft AI Agents Hackathon 2025, Best Overall, $20,000): Supply chain risk analysis using coordinated agents.
- **Apollo Deep Research Meta Agent** (Microsoft, Best Agent in C#): Meta-agent orchestrating research across sources.
- **Likeminds** (Agno Global Agent Hackathon, Best Overall): Agentic multi-social semantic network.

### Pattern 2: Domain-Specific Agent Solutions
Winners pick a specific domain they understand deeply and build agents tailored to it.

**Examples:**
- Healthcare coordination agents
- Legal document analysis agents
- Supply chain monitoring agents
- Fair hiring/recruitment agents

### Pattern 3: "Startup in a Box" Autonomous Agents
Full-lifecycle autonomous agents that can manage complex real-world workflows end-to-end.

**Example:**
- **FoundrAI** (Consensus Hong Kong 2026, 1st place): An autonomous AI agent that manages the entire lifecycle of a startup project, including hiring human developers.

### Pattern 4: Agent + Real-World Action
Agents that don't just analyze but take concrete actions -- making payments, executing trades, deploying code, sending communications.

**Example:**
- **World of Geneva** (SF Agentic Commerce Hackathon, 1st place, $50K+ prize pool): AI agents that autonomously play an MMORPG, completing quests, fighting, and trading.

### Pattern 5: Security-Embedded Agents
Teams that embed security checks, guardrails, and safety mechanisms directly into agent workflows stand out, especially at security-focused events like RSAC.

### What Google Cloud AI Hackathon Winners Did (270,000 developers)
- Focused on practical, real-world impact
- Problem statements mapped to government and industry priorities
- Demonstrated production-ready tooling that plugs into existing enterprise pipelines
- Provided auditability and scalability

## 5. Demo & Presentation Best Practices

### The Golden Rules of Hackathon Demos

**1. Problem First, Solution Second**
- Spend the first 20-30 seconds making the audience feel the pain of the problem
- "Every day, SOC analysts spend 4 hours manually triaging alerts. 80% are false positives."
- The more they grasp the problem, the more they appreciate your solution

**2. One Linear Path, Not a Feature Tour**
- Follow a single user journey from start to finish
- Judges don't need to see everything; they need to see the parts that validate your project's logic
- Example flow: "Alert comes in -> Agent analyzes -> Agent investigates -> Agent recommends action -> Human approves -> Done"

**3. The 3-Minute Rule**
- Hackathon presentations typically last 3-5 minutes
- Every second must reinforce meaning -- no background narratives, extended introductions, or deep process dives
- Practice until you can hit your time naturally without rushing

**4. Live Demo > Slides**
- A working demo is 10x more impressive than slides describing what it could do
- BUT have a video backup in case the live demo fails
- Pre-load data, pre-configure the environment, have everything ready to go

**5. The "Wow Moment"**
- Engineer one memorable moment -- the thing that makes judges look up from their phones
- This is usually the agent doing something unexpectedly intelligent or useful
- Place it in the middle or just past the middle of your demo

### Presentation Structure Template
```
[0:00-0:30]  The Problem (make them feel the pain)
[0:30-1:00]  Our Solution (one sentence + architecture overview)
[1:00-3:00]  Live Demo (single user journey with wow moment)
[3:00-3:30]  Impact & What's Next (metrics, scalability, vision)
[3:30-4:00]  Q&A buffer
```

### Delivery Tips
- Speak clearly and confidently; use simple language
- Maintain eye contact with judges, not your screen
- Assign one dedicated presenter (not everyone talking over each other)
- End slightly early with a crisp message rather than cramming extra content
- Have one team member ready to handle Q&A on technical details

### Submission Essentials
- **Working demo** (required at most hackathons)
- **2-3 minute demo video** (even if presenting live -- many judges review videos later)
- **Screenshots** in your project write-up
- **Clear README** with setup instructions and architecture diagram
- Document as you build, not after

## 6. Common Pitfalls That Lose Hackathons

### The Seven Deadly Sins of Hackathons

**1. Overscoping (The #1 Killer)**
- Trying to build a product that would take months in 24-48 hours
- If you can't demo it clearly in 3 minutes, it's too complex
- **Fix**: Start with ONE core feature. Expand only if you finish early.

**2. Weak or Overly Technical Pitch**
- Engineers tend to talk about implementation details, not impact
- Judges don't care about your database schema -- they care about what it does for users
- **Fix**: Lead with the problem and user impact. Save technical details for Q&A.

**3. Building Before Planning**
- Jumping into code without agreeing on scope, architecture, or who does what
- **Fix**: Spend the first 30-60 minutes planning. Define the MVP, assign roles, agree on tech stack.

**4. Ignoring the Demo Until the Last Hour**
- Building features but not preparing a coherent demo flow
- **Fix**: Build demo-first. Your demo IS your product at a hackathon.

**5. Not Meeting Basic Requirements**
- Failing to use required APIs, missing submission deadlines, incomplete project write-up
- **Fix**: Read the rules document THREE times. Check off every requirement.

**6. "It Works on My Machine" Syndrome**
- Demo crashes on the presentation laptop, API keys expired, dependencies missing
- **Fix**: Test your demo on a fresh environment. Have video backup. Pre-load everything.

**7. Ignoring Sponsor Prizes**
- Only going for the grand prize when sponsor prizes have much better odds
- **Fix**: Intentionally integrate 2-3 sponsor APIs. Apply for every relevant prize category.

### AI-Specific Pitfalls

**8. Hallucination Without Guardrails**
- Agent makes stuff up during the demo in front of judges
- **Fix**: Add validation layers, constrain outputs, use structured generation, have fallback responses.

**9. "It's Just a Wrapper"**
- Thin wrapper around an LLM API call with no real agent logic
- **Fix**: Add tool use, multi-step reasoning, memory, state management -- things that make it a real agent.

**10. Slow Agent Response Times**
- Agent takes 60+ seconds to respond during a 3-minute demo
- **Fix**: Pre-compute what you can. Use streaming. Cache common queries. Optimize your prompts.

## 7. Maximizing Sponsor API Usage

### The Sponsor Prize Strategy

Sponsor prizes are the **highest expected-value play** at any hackathon. Here's why:
- Grand prize: 1 winner out of 50-200 teams (0.5-2% odds)
- Sponsor prize: Often 1 winner out of 5-20 teams that used their API (5-20% odds)
- Many teams ignore sponsor prizes entirely, further improving your odds

### The "Prize Stacking" Approach
1. **Identify all sponsor prize categories** before you start building
2. **Design your architecture** to meaningfully integrate 2-3 sponsor APIs
3. **Apply for every prize category** your project qualifies for
4. This 10x's your odds of winning something

### How to Use Sponsor APIs Like a Winner

**Before the Hackathon:**
- Read sponsor API documentation ahead of time
- Build small proof-of-concepts with each API you plan to use
- Understand each sponsor's **unique selling proposition** (USP) -- what makes their tool different from competitors

**During the Hackathon:**
- Visit the **API Evangelist / Sponsor Booth** early -- they share features you didn't know about and can save you hours
- Start a relationship with sponsor reps; have them teach you about their API
- A sit-down session with the evangelist gets your app tagged with that API, which is additional validation during judging
- Ask sponsors: "What kind of project would impress your team most?"

**In Your Demo:**
- **Explicitly call out** each sponsor API you used and WHY you chose it
- Show how the sponsor's tool was essential (not just bolted on)
- Use sponsor terminology and brand language
- Demonstrate you understand the sponsor's USP

### Sponsor API Integration Depth Levels

| Level | Description | Judge Impression |
|-------|-------------|-----------------|
| **Surface** | "We used X for authentication" | Forgettable |
| **Functional** | "X powers our core agent memory" | Good |
| **Deep** | "We leveraged X's unique feature Y to solve Z in a way that wouldn't be possible with alternatives" | Winner |
| **Showcase** | "Our entire architecture demonstrates why X's approach to agents is superior" | Sponsor's favorite |

## 8. Tactical Playbook: The 24-Hour Gameplan

### Pre-Hackathon (Days Before)
- [ ] Research judges' backgrounds and what they value
- [ ] Read ALL sponsor API docs and build small POCs
- [ ] Prepare your development environment (boilerplates, dependencies, API keys)
- [ ] Draft a 1-paragraph pitch for 3 different project ideas
- [ ] Research past winners of this specific hackathon/organizer
- [ ] Prepare demo recording setup (screen recorder, camera)

### Hour 0-1: Plan & Align
- [ ] Listen to opening talks -- they often drop hints about what judges want
- [ ] Visit sponsor booths and ask what would impress them
- [ ] Finalize your project idea based on intel gathered
- [ ] Define the absolute minimum viable demo (what's the ONE thing it must do?)
- [ ] Assign roles: Builder 1, Builder 2, Demo/Pitch person, Integration person

### Hour 1-8: Build Core Agent
- [ ] Get the core agent loop working end-to-end (even if ugly)
- [ ] Integrate primary sponsor API early
- [ ] Focus on the demo path -- don't build features you won't show
- [ ] Test the happy path continuously

### Hour 8-16: Integrate & Polish
- [ ] Add secondary sponsor API integrations
- [ ] Build the demo UI (can be simple but must be clean)
- [ ] Add error handling for demo scenarios
- [ ] Start writing the project description

### Hour 16-20: Demo Prep
- [ ] Build your demo script (exact steps, exact words)
- [ ] Practice the demo 3+ times
- [ ] Record a backup video
- [ ] Prepare for common judge questions
- [ ] Write the project submission (README, screenshots, description)

### Hour 20-24: Polish & Submit
- [ ] Final demo rehearsal
- [ ] Test on presentation laptop/projector
- [ ] Submit before the deadline (not AT the deadline)
- [ ] Rest for 30 minutes before presenting

### The Winning Mindset
> "A mediocre project with an amazing pitch beats an amazing project with a mediocre pitch."
>
> "Before every decision, ask: Will judges care about this? If no, don't build it."
>
> "Think like a judge: they're tired, they've seen 40 demos, and they want to be impressed in 3 minutes."

## Key Takeaways

1. **Solve a real problem** in a specific domain you understand
2. **Build multi-agent systems** with clear agent roles and coordination
3. **Make it work end-to-end** -- completeness beats feature breadth
4. **Integrate sponsor APIs deeply** and apply for every relevant prize
5. **Nail the demo** -- one linear path, one wow moment, under 3 minutes
6. **Show, don't tell** -- live demo with fallback video
7. **Make your agents explainable** -- transparency wins trust
8. **Document as you go** -- be ready to submit when time is called
9. **Practice your pitch** until it flows naturally
10. **Think like a judge** with every decision you make
